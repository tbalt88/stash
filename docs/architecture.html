Stash — Architecture

One place your agent connects to see all your company data. Input pipes (sources) → output pipes (CLI, MCP, API, UI). This documents the information hierarchy, sharing model, database schema, and API as of the Cartridges / user-scoped redesign.

1. Information hierarchy

Everything is scoped to a user (one implicit, private workspace per user — hidden from the UI; it remains the internal scoping key). Within it:

user └─ workspace (implicit; hidden; the scoping key) ├─ Agents chat over all sources; primary use is via your own agent (CLI/MCP/API) ├─ Sources what the agent can read │ ├─ Files (native) folders + pages + uploaded files │ ├─ Sessions (native) agent transcripts, grouped into session folders │ ├─ GitHub (connected) repos exposed as a filesystem │ ├─ Google Drive (connected) index only; document data fetched lazily │ ├─ Notion (connected) index only; page/db data fetched lazily │ ├─ Slack (connected) full copy via webhook │ └─ Granola (connected) full copy via webhook/pull └─ Cartridges read-only, disposable, curl-able bundles (PLG sharing)

Native sources are workspace-scoped (owned by the user). Connected sources are user-scoped (only their owner sees them). The file system is "just another source."

2. Sharing & permissions

Private by default. A user owns everything in their workspace (workspace membership == ownership). Nobody else sees anything until a shares row exists.

The two-table model

  • shares — grants a principal (a user, or a cartridge) access to an object. This is the primary, team-collaboration path: share a folder/file/session with a person by email. Folder & session-folder shares cascade to their contents.
  • cartridges + cartridge_members + cartridge_items — a read-only bundle (what it contains + who can open it). Secondary, PLG: a link you hand to an agent (curl-able). Adding a source to a cartridge copies a snapshot in so it's self-contained.

Access resolution — check_access(object, user)

owner (user is the workspace member)                         → read + write
OR a user share on the object or an ancestor folder         → per share (read/write)
OR the object is in a cartridge the user can open            → read only
else                                                         → denied

A cartridge is "openable" if it is public/unlisted, owned by the user, or the user is a cartridge member. Cartridges never grant write.

4. The agent surface

The /ask agent (a direct Anthropic tool-use loop) reads across sources through one source-aware tool set, scoped to _current_user() + the sharing rules above:

list_sources()                  native (files, sessions) + the user's connected sources
list_source(source, path?)      navigate a source like a file system
read_source(source, ref)        read one document (lazy-fetched for Drive/Notion)
search(query, source?)          FTS + vector across sources, owner/share scoped
