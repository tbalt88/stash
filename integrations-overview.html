Stash Integrations

How authentication, the connect UI, and data flow work across every source the agent can read.

The agent reads from sources. Two are native to every workspace — Files and Session transcripts. The rest are connected sources: third-party services a user links to their own account. Every connected source is user-scoped — only the person who connected it can see or read it, even within a shared workspace.

1 · How authentication works

Integrations authenticate one of three ways. The connect flow differs, but the result is identical: a credential encrypted at rest, owned by the user, used by the server on their behalf.

OAuth

Standard redirect-and-consent. The user is sent to the provider, approves, and is sent back.

GitHub · Google · Notion · Slack · Jira · Asana

MCP OAuth

OAuth 2.0 over an MCP server with Dynamic Client Registration + PKCE — no pre-shared client secret.

Granola

API key

No redirect. The user pastes credentials into an in-app form; the server validates them upstream.

Gong · Snowflake

The OAuth redirect flow (GitHub, Google, Notion, Slack, Jira, Asana)

  1. User clicks Connect. The app calls GET /api/v1/integrations/{provider}/connect with its bearer token and gets back an authorize_url.
  2. The browser navigates to the provider's consent screen. The request carries an encrypted state blob.
  3. User approves. The provider redirects to …/integrations/{provider}/callback?code=…&state=….
  4. The backend decrypts state to recover the user identity and verify the request (CSRF), then exchanges the code for tokens.
  5. Tokens are encrypted and stored; the user lands back in Settings, now connected.
Why state matters. It's a Fernet-encrypted token carrying {user_id, provider, nonce, return_to} with a 10-minute TTL. Fernet's built-in HMAC gives CSRF protection for free, and it lets the callback — which the browser hits with no auth header — recover who the user is.

The API-key flow (Gong, Snowflake)

  1. User clicks Connect; an inline form appears, built from fields the provider declares (e.g. Gong's Access Key + Secret).
  2. The app POSTs the values to …/integrations/{provider}/credentials.
  3. The provider validates them against the upstream API (Gong → /v2/workspaces; Snowflake → SELECT CURRENT_USER()). Bad credentials return a clean 400.
  4. On success the credential bundle is stored — JSON-encoded into the same encrypted column OAuth tokens use, with no expiry.

Storage & security (all three models)

  • Encrypted at rest. Every access/refresh token (and every pasted credential bundle) is encrypted with Fernet using a single server key, INTEGRATIONS_ENCRYPTION_KEY.
  • Refresh-on-use. For providers that issue refresh tokens (Google, Jira, Asana, Granola), the server refreshes automatically when a token is within 60s of expiry and writes the rotated token back — so Atlassian/Asana token rotation is transparent.
  • User-scoped. A credential belongs to one (user_id, provider). Sync and reads run server-side on behalf of that owner; no other member can use or see it.
  • Provider-agnostic core. One OAuth router, one storage layer, one set of agent tools serve every provider — adding a source doesn't touch them.

Connect states

  • Not connected, OAuth → Connect redirects to the provider.
  • Not connected, API key → Connect reveals the inline credential form shown above.
  • Connected, picker → Add repo / page / project opens a searchable list.
  • Connected, workspace-wide → Add creates the source in one click (Slack, Granola, Gong, Snowflake).
  • Misconfigured server → Unavailable with a tooltip explaining what env var is missing.

Pickers

After connecting, sources that have many “things” show a searchable picker so the user chooses exactly what to expose:

  • GitHub → repositories
  • Notion → pages & databases shared with the integration
  • Jira → projects (across all Atlassian sites)
  • Asana → projects (across all workspaces)

Connected sources list

Below the connectors, each added source appears with its name and a Sync (re-index now) and Remove action. Snowflake appears here too, but it has no picker or browse view — the agent uses it through query tools rather than the UI.

3 · How data works

What happens to a source's data depends on its capability. There are three, and they map to three storage strategies.

navigable

Browsed like a file system (list_source → read_source). Repos, Drive folders, Notion trees, Asana projects.

searchable

A flat pool searched by full text (search). Slack/Granola messages, Jira issues, Gong calls.

queryable

Queried live with read-only SQL (query_source). Snowflake only — nothing is copied or indexed.

Two storage strategies for documents

StrategyWhat's storedHow a read resolvesSources
Copied content The document body, a content hash, and a vector embedding — in a per-integration table. Returned straight from the table. Full-text search + embeddings work directly. GitHub, Slack, Granola, Jira, Asana, Gong
Index-only Just the path/name + the provider's reference id — never the body. The body is fetched lazily from the provider at read time, with the owner's token. Google Drive, Notion
None (live) Nothing. No table, no index. The agent runs a read-only SQL query against the warehouse on demand. Snowflake

How data stays fresh (sync)

Pull sources

A scheduler periodically finds sources whose sync is due and runs their indexer, which crawls the provider and upserts documents. Re-syncs are idempotent — unchanged docs are skipped (content-hash), and docs that disappeared upstream are soft-deleted.

GitHub · Drive · Notion · Jira · Asana · Gong

Push sources

New activity streams in via webhooks as it happens, so the data is near-real-time. A periodic backfill runs as a safety net to catch anything missed.

Slack · Granola

How the agent reaches it

All sources — native and connected — sit behind one uniform toolset, so the agent treats a GitHub repo, a Notion tree, and a Snowflake warehouse the same way until it needs their specific capability:

# discover what's available list_sources() → files, sessions, + the user's connected sources (each with a capability) # navigable / searchable list_source(source, path) → entries, like ls read_source(source, ref) → one document's content search(query, source?) → full-text across one source or everything # queryable (Snowflake) query_source(source, sql) → runs read-only SQL, returns capped rows
Snowflake is read-only by construction. Only SELECT / WITH / SHOW / DESCRIBE / EXPLAIN are allowed; multiple statements (a piggy-backed ; DROP …) are rejected, table identifiers are validated against injection, and result rows are capped. Pairing it with a read-only Snowflake role is the recommended second layer.
